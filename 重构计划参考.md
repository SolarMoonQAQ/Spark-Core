# Spark-Core 动画与模型动态加载重构计划参考

1. 引言

1.1. 目标
为 Spark-Core 模组实现动画与模型的动态加载能力，允许在游戏运行时加载新的或更新现有的动画与模型资源，而无需重启游戏。

1.2. 当前挑战
Minecraft Forge/NeoForge 的注册系统主要设计为在游戏初始化阶段完成所有内容的注册。一旦注册阶段结束，直接向其动态添加新的注册条目（尤其是对于像 `TypedAnimation` 这样已经有自定义注册表系统的对象）是困难的，甚至不可行。

1.3. 核心思路概述
本文将探讨两种主要的重构思路：
1.  **访问点拦截**：通过修改资源获取逻辑，优先查找动态资源，实现对现有注册流程的最小侵入。
2.  **自定义注册表**：通过创建一个能感知动态条目的自定义注册表实现，从更底层支持动态性。

2. 理解当前的静态资源加载与注册机制

2.1. 类型化动画 (TypedAnimation)
* **定义与注册**:
    * `SparkTypedAnimations.kt`: 定义了诸如 `IDLE`, `WALK` 等静态的 `TypedAnimation` 实例。
        ```kotlin
        // src/main/kotlin/cn/solarmoon/spark_core/registry/common/SparkTypedAnimations.kt
        @JvmStatic
        val IDLE = createStateAnim("idle")
        @JvmStatic
        val WALK = createMoveStateAnim("walk")
        // ...
        fun createStateAnim(name: String, index: ResourceLocation = ResourceLocation.withDefaultNamespace("player"), provider: TypedAnimProvider = {}) = SparkCore.REGISTER.typedAnimation()
            .id(name)
            .animIndex(AnimIndex(index, "EntityState/$name"))
            .provider(provider)
            .build()
        ```
    * 这些动画通过 `SparkCore.REGISTER.typedAnimation()` （返回 `TypedAnimationBuilder`）构建，并注册到 `SparkRegistries.TYPED_ANIMATION` 这个自定义的 NeoForge 注册表中。
        ```kotlin
        // src/main/kotlin/cn/solarmoon/spark_core/entry_builder/ObjectRegister.kt
        val typedAnimationDeferredRegister = lazy { DeferredRegister.create(SparkRegistries.TYPED_ANIMATION, modId) }
        // ...
        fun typedAnimation(): TypedAnimationBuilder {
            if (!typedAnimationDeferredRegister.isInitialized()) {
                typedAnimationDeferredRegister.value.register(modBus!!)
            }
            return TypedAnimationBuilder(typedAnimationDeferredRegister.value)
        }
        ```
* **结构**:
    * `TypedAnimation.kt`: 每个实例包含一个 `AnimIndex` (动画索引，指向具体的模型资源和动画名称) 和一个 `provider` (动画事件处理器)。
        ```kotlin
        // src/main/kotlin/cn/solarmoon/spark_core/animation/anim/play/TypedAnimation.kt
        class TypedAnimation(
            val index: AnimIndex,
            private val provider: TypedAnimProvider
        ) {
            val id get() = SparkRegistries.TYPED_ANIMATION.getId(this) // 从注册表获取ID
            val registryKey get() = SparkRegistries.TYPED_ANIMATION.getKey(this) // 从注册表获取Key
            // ...
        }
        ```
    * `AnimIndex`: 包含模型的 `ResourceLocation` 和动画的 `name`。

## 3. 重构方案一：通过拦截资源访问实现动态加载与回退

核心思路：不直接修改 NeoForge 的注册流程，而是在代码**请求**模型或动画数据时进行拦截。

### 3.1. 动态资源存储
首先，需要为动态加载的资源创建新的存储容器。这些可以是简单的 `Map` 结构，由一个新的动态资源管理器来维护。

```kotlin
// 建议新建一个管理类，例如 DynamicAssetManager.kt
object DynamicAssetManager {
    val dynamicModels = java.util.concurrent.ConcurrentHashMap<ResourceLocation, OModel>()
    val dynamicAnimationSets = java.util.concurrent.ConcurrentHashMap<ResourceLocation, OAnimationSet>()
    // (可选) 如果需要完全动态的 TypedAnimation 类型
    val dynamicTypedAnimations = java.util.concurrent.ConcurrentHashMap<ResourceLocation, TypedAnimation>()

    // 此处应包含加载、卸载、重载动态资源的逻辑，
    // 例如从特定文件夹读取文件，解析并填充上述 Map
    // 参考之前聊天记录中提到的 DynamicAnimationModelLoader
}
```


3.2. 修改资源获取的入口点
3.2.1. 原始模型 (OModel) 的获取
当前访问方式: 代码可能直接通过 OModel.ORIGINS[id] 或类似的静态访问来获取模型。
修改建议: 封装 OModel 的获取逻辑。
在 OModel.kt 的伴生对象中或创建一个新的工具类 ModelAccessor.kt。
// 在 OModel.kt 的 companion object 中修改或新增
companion object {
    @JvmStatic
    val ORIGINS = linkedMapOf<ResourceLocation, OModel>() // 保留静态资源
    @JvmStatic
    val EMPTY get() = OModel(0, 0, linkedMapOf())

    // 新增的获取方法
    @JvmStatic
    fun getModel(id: ResourceLocation): OModel? {
        // 1. 优先从动态缓存获取
        DynamicAssetManager.dynamicModels[id]?.let { return it }
        // 2. 若动态缓存没有，则从静态 ORIGINS 获取
        return ORIGINS[id]
    }
}



影响: 所有之前直接访问 OModel.ORIGINS 的地方，都需要改为调用 OModel.getModel(id)。
3.2.2. 原始动画集 (OAnimationSet) 的获取
当前访问方式: 例如在 TypedAnimation.exist() 中通过 OAnimationSet.get(index.index)。
修改建议: 直接修改 OAnimationSet.get() 方法。
// src/main/kotlin/cn/solarmoon/spark_core/animation/anim/origin/OAnimationSet.kt
companion object {
    @JvmStatic
    var ORIGINS = linkedMapOf<ResourceLocation, OAnimationSet>() // 保留静态资源
    val EMPTY get() = OAnimationSet(linkedMapOf())

    // 修改此方法
    @JvmStatic
    fun get(id: ResourceLocation): OAnimationSet {
        // 1. 优先从动态缓存获取
        DynamicAssetManager.dynamicAnimationSets[id]?.let { return it }
        // 2. 若动态缓存没有，则从静态 ORIGINS 获取
        return ORIGINS[id] ?: EMPTY // 如果都没有，返回 EMPTY
    }

    // (可选) 如果 OAnimationSet.ORIGINS 被其他地方直接引用，也需要考虑拦截
}



影响: 此修改将透明地影响所有通过 OAnimationSet.get() 获取动画集的代码，包括 TypedAnimation 内部的逻辑。
3.2.3. 类型化动画 (TypedAnimation) 的获取与处理
这部分相对复杂，取决于动态化的程度。
情况 A: 动态更新现有 TypedAnimation 类型的数据
如果只是希望 IDLE, WALK 等已注册的 TypedAnimation 能使用动态加载的模型/动画文件，那么上述对 OAnimationSet.get() 和 OModel.getModel() 的修改可能已经足够。因为 TypedAnimation 通过其 AnimIndex 最终会解析到这些原始数据。
AnimIndex 包含 ResourceLocation (用于模型) 和 name (用于动画集内的具体动画)。当 OAnimationSet.get() 返回的是动态加载的 OAnimationSet 时，TypedAnimation 就能使用到动态的动画数据。
情况 B: 创建全新的、动态定义的 TypedAnimation 类型
这意味着 TypedAnimation 对象本身是动态创建和管理的，而不是预先注册到 NeoForge 的 SparkRegistries.TYPED_ANIMATION。
入口点: 需要修改通过 ResourceLocation 查找 TypedAnimation 的地方。一个关键例子是 PlayerStateAnimMachine.kt：

// ...
// SparkRegistries.TYPED_ANIMATION.get(ResourceLocation.parse(sName))?.let { // 旧代码
// ...



修改建议: 创建一个新的访问器函数，用于统一查询动态和静态的 TypedAnimation。
// 在 DynamicAssetManager 或新建的 TypedAnimationAccessor.kt 中
fun getEffectiveTypedAnimation(id: ResourceLocation): TypedAnimation? {
    // 1. 优先从动态 TypedAnimation 缓存获取
    DynamicAssetManager.dynamicTypedAnimations[id]?.let { return it }
    // 2. 若动态缓存没有，则从 NeoForge 注册表获取
    return SparkRegistries.TYPED_ANIMATION.get(id)

然后更新调用点：

4. 备选重构方案二：通过自定义注册表实现动态加载 (DynamicAwareRegistry)
这个方案更为深入，旨在从注册表层面直接支持动态性，使得代码在查询注册表时能无缝访问到动态和静态的资源。
4.1. 核心思想：DynamicAwareRegistry
创建 DynamicAwareRegistry<T>:
这是一个新的类，它实现了 NeoForge 的 Registry<T> 接口。
内部它会持有一个标准的 NeoForge Registry<T> 实例（用于存储和管理通过 DeferredRegister 静态注册的条目）。
同时，它会维护一个独立的、用于动态添加和移除条目的数据结构（例如一个 java.util.concurrent.ConcurrentHashMap<ResourceLocation, T>）。
DynamicAwareRegistry 将重写 Registry<T> 接口中的关键查找方法，如 get(ResourceLocation name)、get(ResourceKey<T> key)、byId(int id)、containsKey(ResourceLocation name)、entrySet()、keySet() 等。
这些重写的方法会首先查询其内部的动态数据结构。如果找不到，则委托给其持有的静态 Registry 实例进行查找。
此类还需要提供新的公开方法，例如 registerDynamic(ResourceLocation name, T value) 和 unregisterDynamic(ResourceLocation name)，这些方法可以在运行时被调用来直接修改动态存储。
修改 RegistryBuilder (Spark-Core 内的自定义版本):
Spark-Core 中的 cn.solarmoon.spark_core.entry_builder.common.RegistryBuilder 的 build 方法当前返回一个标准的 Registry<T>。
需要修改此 build 方法，使其返回一个 DynamicAwareRegistry<T> 的实例。
具体做法是：
原有的逻辑（通过 NeoForge 的 RegistryBuilder）创建标准的 Registry<T> (这将作为 DynamicAwareRegistry 的静态部分)。
然后，build 方法创建一个 DynamicAwareRegistry<T> 实例，并将前面创建的标准 NeoForge Registry 作为构造参数传入（用于代理静态条目）。
返回这个新建的 DynamicAwareRegistry<T> 实例。
修改 ObjectRegister 和 SparkRegistries:
确保 SparkRegistries.TYPED_ANIMATION (以及其他任何需要动态能力的注册表，如模型或动画集的注册表，如果它们也通过这种方式管理) 被初始化为此 DynamicAwareRegistry<TypedAnimation> 的实例。
ObjectRegister.typedAnimation() 返回 TypedAnimationBuilder，后者内部持有 DeferredRegister<TypedAnimation>。
DeferredRegister 在 RegisterEvent 期间会将收集到的条目注册到其关联的 Registry 中。在这个方案下，这些静态注册的条目将被注册到 DynamicAwareRegistry 内部持有的那个标准 NeoForge Registry（即静态部分）。
4.2. TypedAnimationBuilder 的角色调整
静态注册时 (Mod 初始化期间):
TypedAnimationBuilder 的现有逻辑 (通过 DeferredRegister.register()) 仍然有效。它会收集在代码中预定义的 TypedAnimation。
这些动画最终会通过 DeferredRegister 注册到 DynamicAwareRegistry 的静态部分。
由于 DeferredRegister.register() 方法有明确的检查，禁止在 RegisterEvent 之后调用，因此不能用此路径进行运行时动态注册。
动态创建与注册时 (游戏运行时):
仍然可以使用 TypedAnimationBuilder 的构建逻辑 (如 .id(), .animIndex(), .provider()) 来方便地创建 TypedAnimation 的实例本身。
但是，在获得 TypedAnimation 实例后，不会通过 DeferredRegister 的流程来注册它。
而是直接调用 DynamicAwareRegistry 实例上的新动态注册方法，例如：
// 假设 SparkRegistries.TYPED_ANIMATION 是 DynamicAwareRegistry<TypedAnimation> 的实例
val typedAnimBuilder = SparkCore.REGISTER.typedAnimation() // 获取 Builder
// ... 使用 builder 配置 name, animIndex, provider ...
val newDynamicAnimName = ResourceLocation.fromNamespaceAndPath("mydynamicmod", "custom_fly_anim")
val newDynamicAnimInstance = typedAnimBuilder
                                .id(newDynamicAnimName.getPath()) // Builder的id方法可能需要调整以适应不通过DeferredRegister的场景
                                .animIndex(AnimIndex(ResourceLocation.parse("mydynamicmod:my_entity_model"), "fly"))
                                .provider { /* ... 事件处理 ... */ }
                                .construct() // 假设Builder有一个方法直接返回TypedAnimation实例而不触发注册

// 通过 DynamicAwareRegistry 的新方法进行动态注册
val typedAnimationRegistry = SparkRegistries.TYPED_ANIMATION as? DynamicAwareRegistry<TypedAnimation>
typedAnimationRegistry?.registerDynamic(newDynamicAnimName, newDynamicAnimInstance)



这意味着 TypedAnimationBuilder 可能需要一个额外的方法（例如 construct() 或 buildObjectOnly())，该方法仅构造 TypedAnimation 对象而不尝试通过 DeferredRegister 注册。
4.3. 优缺点分析 (自定义注册表方案)
优点:
查询接口统一和透明: 代码中任何地方通过 SparkRegistries.TYPED_ANIMATION.get(id) 或 .byId(payload.animId) 进行查询时，将自动获得动态感知能力，无需修改大量查询点。
中央控制: 动态行为被封装在 DynamicAwareRegistry 内部，逻辑更集中。
缺点/挑战:
Registry<T> 接口的复杂性: Registry<T> 接口非常庞大且复杂。DynamicAwareRegistry 需要正确实现或代理所有这些方法（包括迭代、keySet, entrySet, holders, tags, 生命周期管理等），确保其行为对于静态和动态条目的组合是一致且正确的。这非常容易出错，并可能引入难以追踪的bug。
生命周期和状态管理: 标准 Registry 在注册后可能会被“冻结”（freeze() 方法）。DynamicAwareRegistry 需要处理好自身的可变性与所包装的静态 Registry 可能的不可变性之间的关系。
线程安全: 动态添加/删除条目需要仔细考虑线程安全问题，尤其是在服务器环境中。所有动态修改操作以及可能的并发读取都需要同步。
网络同步: 如果这些动态注册的 TypedAnimation 需要在客户端和服务器之间同步其存在性或ID，DynamicAwareRegistry 需要配合网络层进行处理。当动态添加一个 TypedAnimation 时，可能需要设计新的网络消息来通知客户端。
与NeoForge生态的兼容性: 需要确保这种高度自定义的 Registry 实现不会破坏 NeoForge 系统中其他依赖于标准 Registry 行为的部分或第三方模组的交互。
实现复杂度高: 相较于拦截访问点的方案，此方案的实现难度和工作量要大得多。
5. 方案对比与选择考量
| 特性 | 方案一 (拦截访问点) | 方案二 (自定义注册表 DynamicAwareRegistry) |
| 实现复杂度 | 较低 | 非常高 |
| 代码侵入性 | 主要在数据获取点，对核心注册流程侵入少 | 深入注册表层面，对 RegistryBuilder, ObjectRegister 有较大改动 |
| 查询透明度 | 需要修改查询点或使用新的访问器 | 查询接口统一，对使用者透明 |
| 风险性 | 较低，改动点相对独立 | 较高，Registry接口复杂，易出错，兼容性风险 |
| 灵活性 | 良好 | 非常好，从根本上支持动态性 |
| 工作量 | 中等 | 大 |
选择建议:
如果项目时间和资源有限，或者希望尽可能降低风险并减少对核心代码的修改，方案一（拦截访问点） 是更务实的选择。它能以相对较小的代价实现核心的动态加载需求。
如果追求更彻底、更“优雅”的解决方案，并且团队有足够的技术实力和时间来应对实现自定义注册表带来的复杂性和风险，那么方案二（DynamicAwareRegistry） 可以提供更完美的动态集成。但务必进行充分的设计和严格的测试。
对于大多数情况，建议从方案一入手，因为它提供了较好的成本效益和较低的风险。如果后续发现方案一的局限性确实无法满足需求，再考虑投入资源实现方案二。
6. 通用注意事项 (适用于任一方案)
引用管理: 确保对动态加载资源的引用是安全的。如果一个动态资源被卸载，持有其引用的地方需要能妥善处理（例如，回退到默认资源）。
性能:频繁的文件IO和解析可能会影响性能。考虑缓存和按需加载。
ID 冲突: 需要有策略来处理动态资源与静态资源，或不同动态资源之间的 ID (ResourceLocation) 冲突问题。动态加载的资源通常应优先。
客户端同步: 确保客户端能够获取到动态加载的资源信息，并正确显示。这对于视觉相关的模型和动画至关重要。
错误处理与日志: 健壮的错误处理机制和详细的日志记录对于调试动态加载问题至关重要。
7. 总结
为 Spark-Core 引入动画和模型的动态加载能力将显著提升其灵活性和可扩展性。通过“拦截访问点”的方案可以在较低
